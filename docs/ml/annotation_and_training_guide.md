# ML Annotation & Training Guide

Короткий систематизированный референс для разметки в CVAT и обучения моделей
`parts` и `damages`. Держите открытым во время разметки.

---

## 0. Общие правила разметки (читайте ПЕРЕД каждой сессией)

1. **Только уверенные метки.** Сомнение ≥ 20% → пропустить объект. 300 чистых
   разметок всегда лучше 600 грязных.
2. **Один разметчик, один стиль.** Если правила меняются — начните новую папку
   `labels_v2/` и переразметьте уже сделанное, не смешивайте стили.
3. **Маска должна плотно облегать объект.** YOLO-seg считает mask loss внутри
   bbox; «большой bbox с маленькой маской внутри» → модель учится предсказывать
   фоновую область как foreground.
4. **Несколько объектов одного класса** → несколько инстансов, не один большой.
5. **Частично видимый объект** → размечается ТОЛЬКО видимая часть, без
   «додумывания» контура.
6. **Класс объекта неоднозначен?** → не размечайте. Лучше пропуск, чем шум.
7. **Каждые ~50 размеченных картинок** — самоаудит: прокрутить подряд, проверить
   когерентность. Правьте расхождения сразу.

## 1. Модель деталей (`parts`)

**Классы (12):** `door, front_fender, rear_fender, trunk, hood, roof, headlight,
front_windshield, rear_windshield, side_window, wheel, bumper`.

**Текущее состояние:** 221 размеченное изображение, сильный перекос
(`bumper`×271 vs `rear_windshield`×21). Нужно добрать редкие классы.

### Правила per-класс

| Класс              | Что размечать                                                         | Что НЕ размечать                                              |
| ------------------ | --------------------------------------------------------------------- | ------------------------------------------------------------- |
| `door`             | Только видимая часть одной двери; у 4-дверных — передняя и задняя раздельно | Молдинги, ручки, стекло двери (это `side_window`)        |
| `front_fender`     | Только крыло, от колёсной арки до капота/двери                        | Бампер, колесо, фара                                          |
| `rear_fender`      | Аналогично — задняя часть борта                                       | Задний бампер, задние фонари, стекло                          |
| `trunk`            | Видимая часть крышки багажника                                        | Задний бампер, задний номер, стекло                           |
| `hood`             | Видимая часть капота                                                  | Решётка, фары, бампер, ветровое стекло                        |
| `roof`             | Крыша (внешняя панель)                                                | Стойки, стёкла                                                |
| `headlight`        | Корпус фары (одна = один инстанс)                                     | Противотуманки, поворотники в бампере — пропускайте           |
| `front_windshield` | Лобовое стекло целиком                                                | Дворники, резинки                                             |
| `rear_windshield`  | Заднее стекло                                                         | Дворник/рамка                                                 |
| `side_window`      | Каждое боковое стекло — **отдельный инстанс**                          | Рамка, молдинг                                                |
| `wheel`            | Колесо целиком (диск + шина), только видимая часть                    | Арка, брызговик                                               |
| `bumper`           | Передний ИЛИ задний бампер целиком                                    | Решётка радиатора, номер, противотуманки                      |

### Приоритеты дозбора данных

По текущему датасету нужно целенаправленно добрать (минимум до 50 картинок
каждого класса):

* `rear_windshield` — **критично** (11 картинок, модель этот класс почти не
  видит).
* `roof` — 26 картинок.
* `trunk` — 35 картинок.
* `side_window` — 39 картинок.

Источники: r/justrolledintotheshop, r/whatcarshouldIbuy, Яндекс.Картинки по
запросам «машина вид сбоку», «машина сзади», «крыша авто повреждение».

### Типичные ошибки (проверяйте перед закрытием картинки)

* Одна маска `door` охватила и дверь, и крыло → разделить.
* `bumper` уехал на колёсные арки → обрезать по линии кузова.
* Капот размечен с решёткой радиатора → убрать решётку.
* У 4-дверного авто обе двери одной маской → разделить на 2 инстанса.

## 2. Модель повреждений (`damages`)

**Классы (8):** `scratch, dent, paint_chip, rust, crack, broken_glass, flat_tire,
broken_headlight`.

**Текущее состояние:** 327 размеченных crop-ов, 230 картинок без разметки (их
нужно либо разметить, либо физически убрать). Перекос: `scratch`×1024 vs
`flat_tire`×13.

### Правила per-класс (ГЛАВНОЕ ИЗМЕНЕНИЕ — перечитайте, даже если размечали раньше)

#### `scratch`

* Каждая **группа близких царапин** — **отдельный инстанс**. Не одна большая
  маска «на все царапины детали».
* Маска — **узкая полоса вокруг самой линии царапины, 2–5 пикселей толщиной**.
  Не обводите пустое пространство между царапинами.
* Если линий много и они идут пучком — один инстанс с несколькими отдельными
  «полосами» допустим, но суммарная площадь маски должна быть малой.
* Bbox маленькой царапины должен быть реально маленьким. Если bbox покрывает
  половину детали — вы размечаете неправильно.

> **Почему это важно:** YOLO-seg считает mask loss внутри bbox. Огромный bbox с
> тонкой царапиной внутри = модель учит, что вся область = «царапина». Отсюда
> ложные срабатывания на половине кузова.

#### `dent` (вмятина)

* Маска = **область деформации поверхности**, не тень и не блик.
* Контур — по границе изменения геометрии (там, где отражение «ломается»).
* Маска **сплошная**, без дыр внутри.
* Мелкие близкие вмятины (<20 px на crop-е) → объединить в один инстанс.
* Крупные раздельные вмятины → отдельные инстансы.

#### `paint_chip` (отвалившийся кусок краски)

* Закрашивайте всю область оголённого металла/грунта.
* Каждый отдельный скол — отдельный инстанс.
* Не путайте с `rust`: если металл чистый — `paint_chip`, если уже рыжий — `rust`.

#### `rust` (ржавчина)

* Закрашивайте всю рыжую/коричневую область целиком.
* Рыжие точки размером < 5 px **не размечайте** (фон).
* Обширные пятна — один инстанс с реальной формой маски.

#### `crack` (трещина) — ПЕРЕСМОТРЕТЬ ВСЕ СТАРЫЕ РАЗМЕТКИ

Относится **только** к:

* Реальным трещинам/разрывам в **материале** (металл прорезан, пластик
  расколот).
* Не относится к:
  * Щели между соседними деталями (это НЕ повреждение, даже если шов «кривой»).
  * Деталям «выпирающим за пределы машины» — это `dent` (деформация), не `crack`.
  * «Треугольникам вне машины» — НЕ размечать вообще.

Маска — **узкая полоса по самой линии разрыва**, не область вокруг.

#### `broken_glass` (разбитое стекло)

* Маска — область с разрушенным стеклом (паутина трещин, пробоины, осколки).
* Целые трещины по стеклу тоже сюда, не в `crack`.
* Один разбитый участок = один инстанс.

#### `broken_headlight` (разбитая фара)

* Маска по корпусу/стеклу фары, где оно разбито/треснуто.
* Отвалившаяся фара (отсутствие фары полностью) — всё равно `broken_headlight`,
  маска по месту, где она была.

#### `flat_tire` (спущенная шина)

* Маска — область деформации шины (там, где она «просела»).
* Это один из редких классов, нужно добрать ≥ 100 примеров. Сейчас 13 — модель
  его не выучит никак.

### Когда НЕ размечать вообще (частые случаи, где все путаются)

* Грязь, пыль, капли воды → фон.
* Блики на полированном кузове → фон.
* Швы между деталями, даже если они «неровные» → фон.
* Старые потёртости, полироль, следы мойки → фон.
* Заводские штампы, логотипы → фон.

### Правило разметки crop-а

* Размечайте повреждение ТОЛЬКО на детали, под которой сделан crop. Если на
  crop-е двери видна часть соседнего крыла — повреждения на крыле **не
  размечать**, они попадут со своего crop-а.

### Приоритеты дозбора данных

* `flat_tire`: +100 картинок (сейчас 13).
* `rust`: +60 (сейчас 42).
* `broken_glass`: +50 (сейчас 55).
* `broken_headlight`: +50 (сейчас 71).

`scratch` **не добирать** — 1024 инстансов достаточно, качество важнее объёма.
Наоборот, при переразметке часть старых «простыней» превратится в корректные
мелкие инстансы — это уже большой выигрыш.

### Откуда брать crop-ы для разметки повреждений

Crop-ы должны быть согласованы с **текущей** моделью деталей. Перегенерация:

```bash
python scripts/crop_parts_for_damage_dataset.py \
    --images-dir <raw_images> \
    --weights test/details2004_learned/best_parts.pt \
    --output test/damages/crops/
```

(Скрипт уже лежит в репозитории — обновляйте веса `--weights` после каждого
переобучения модели деталей.)

## 3. Как организован датасет после всех правок

Оба датасета используют «file-list» формат YOLO (непередвигаемые файлы + `.txt`
списки):

```
test/
├─ parts/
│   ├─ images/          ← все фото (физически не перемещаются)
│   ├─ labels/          ← все .txt разметки
│   ├─ splits/
│   │   ├─ train.txt    ← пути к train-картинкам
│   │   ├─ val.txt
│   │   └─ test.txt
│   └─ data.yaml        ← ссылается на splits/*.txt
└─ damages/
    └─ ... (та же структура)
```

Сейчас файлы лежат в старых папках (`test/images_details/Train`,
`test/0-604damages/...`). Их трогать не нужно — скрипты работают с абсолютными
путями через file-listы.

Итоговые YAML создают скрипты `scripts/ml/split_dataset.py`. См.
`scripts/ml/README.md`.

## 4. Гиперпараметры обучения (baseline)

Эти значения подобраны под ваш объём данных (200–400 картинок), не меняйте их
без причины.

### Общие для обеих моделей

| Параметр        | Значение            | Пояснение                                              |
| --------------- | ------------------- | ------------------------------------------------------ |
| `model`         | `yolov8m-seg.pt`    | `l`/`x` переобучатся на вашем объёме.                  |
| `epochs`        | 150                 | С ранней остановкой `patience=30`.                     |
| `imgsz`         | 640                 | Меньше — теряем мелкие повреждения, больше — медленнее.|
| `batch`         | 16 (A100) / 8 (T4)  | Главное — стабильный BN.                               |
| `optimizer`     | `AdamW`             | SGD нестабилен при малом батче.                        |
| `lr0`           | 0.001               | `AdamW` не любит высокий LR.                           |
| `weight_decay`  | 0.0005              |                                                        |
| `cos_lr`        | `True`              | Косинусный scheduler даёт мягкое затухание.            |
| `patience`      | 30                  | Early stop по val mAP.                                 |
| `close_mosaic`  | 15                  | Последние 15 эпох — без mosaic (model «приземляется»). |

### Аугментации

| Параметр       | Значение | Комментарий                                           |
| -------------- | -------- | ----------------------------------------------------- |
| `hsv_h`        | 0.015    |                                                       |
| `hsv_s`        | 0.7      |                                                       |
| `hsv_v`        | 0.4      |                                                       |
| `degrees`      | 8        | Фото машин близки к горизонту.                        |
| `translate`    | 0.1      |                                                       |
| `scale`        | 0.5      |                                                       |
| `shear`        | 0        | Не искажать геометрию машины.                         |
| `perspective`  | 0.0005   |                                                       |
| `fliplr`       | 0.5      |                                                       |
| `flipud`       | 0        | Перевёрнутых машин в проде не бывает.                 |
| `mosaic`       | 1.0      |                                                       |
| `mixup`        | 0        | Ломает маски.                                         |
| `copy_paste`   | 0.3      | **Ключ к борьбе с дисбалансом** для `seg`.            |
| `cls_pw`       | 0.25     | Степень inverse-frequency весов классов (0=выкл., 1=полный). |

### Только для `damages`

Так как `scratch` доминирует 4:1 по инстансам, включите:

* `cls_pw=1.0` (полные inverse-frequency веса; сильнее, чем у `parts`).
* Дополнительно — **oversampling** редких классов через переиспользование
  картинок в `train.txt` (скрипт сплита поддерживает `--oversample`).

### Пороги уверенности после обучения

Не берите наобум. Используйте `results.csv` и confusion matrix:

1. Посчитайте F1 per-class на val по сетке порогов `[0.15, 0.20, …, 0.60]`.
2. Возьмите порог, максимизирующий F1 у **самого плохого** класса (ваш bottleneck).
3. В проде этот порог идёт в `ml_thresholds.py`.

После правильного переобучения ожидаемые рабочие пороги:

* `parts`: 0.40–0.50.
* `damages`: 0.30–0.45.

Если в проде работают пороги 0.15–0.20 — модель сырая, не применяйте её, а
возвращайтесь в CVAT.

## 5. Как читать метрики обучения

В `runs/segment/<name>/results.csv` смотрите по порядку:

1. **`metrics/mAP50(B)` и `mAP50(M)`** — bbox и mask, val. Должны расти
   монотонно первые ~50 эпох, потом выходить на плато.
2. **`val/box_loss`, `val/seg_loss`, `val/cls_loss`** — должны **падать**. Если
   `val/*` падает, потом **растёт** — переобучение, early stop должен сработать.
3. **`results/confusion_matrix.png`** — главный инструмент диагностики.
   Диагональ = правильные; строки = «что я сказал», столбцы = «что было на
   самом деле». Частая проблема — целая строка уходит в класс «background» →
   модель просто не видит этот класс.
4. **`results/BoxPR_curve.png` и `MaskPR_curve.png`** — площадь под кривой =
   mAP. Резкое падение precision при росте recall → модель генерит много FP.

**Красные флаги:**

* Train loss быстро падает, val loss стоит → переобучение (мало данных,
  агрессивные гиперпараметры).
* mAP = 0 для какого-то класса → у класса <20 инстансов в train, либо вообще
  нет в val.
* `cls_loss` в val не падает → модель не различает классы (аннотация грязная
  или классы семантически неотличимы).

## 6. Процесс в долгую

Один раз организованный цикл важнее разовой отличной модели:

1. **Каждый спорный кейс в проде** — в `data/hard_cases/` (картинка + отчёт
   модели).
2. **Раз в месяц** — доразметить `hard_cases`, влить в основной датасет,
   переобучить.
3. **Не удалять старые веса.** Храните `best_parts_v1.pt`, `best_parts_v2.pt`
   и т.д., чтобы при регрессии можно было быстро откатиться.
4. **Версионируйте `data.yaml`** — коммитьте в git. Это ваш единственный
   человекочитаемый артефакт конфигурации датасета.
